Архитектура монолитная.  

## Ключевые принципы

* Нельзя допускать ошибок первого рода (ложно положительный результат); 
* Каждая рекомендация должна четко ссылаться на существующий документ без потери точности информации, следовательно, каждое утверждение должно быть подкреплено данными (клиническими рекомендациями или предоставляемыми документами);
* При отсутствии заболевания выдавать советующие сообщение об ошибке (см предыдущий пункт);
* Чтобы четко отвечать на вопросы пользователя данные должны быть предобработаны, размечены;
* Один и тот же запрос должен давать один и тот же ответ;
* Возможность простого удаление или добавления новых источников.

Для Backend разработки будет использоваться язык программирования python и его библиотеки, фреймворк FastAPI.

## Взаимодействие API

Для взаимодействия между fronted и backend можно воспользоваться WebSocket / Server-Sent Events (SSE). ML модель может изначально загружаться в память при старте FastAPI. Контекст будет сохраняться во время действия сессии.

Для удобства пользователя было бы неплохо чтобы было исправление опечаток, например PySpellChecker или Pymorphy2 (но он может не знать редких слов). Приведение к нижнему регистру. Также из текста запроса необходимо извлекать ключевые слова, медицинские термины и классифицировать намерения (лечение, диагностика, профилактика). Дальше поиск соответствующих фрагментов текста необходимых для ответа на запрос. Формирование ответа и передача его пользователю (можно передавать JSON файлы). Перед отправкой надо проверять правильно ли указан документ и не сгенерировался ли он.

## Python/FastAPI/PyDantic

FastAPI представляет быстрый высокопроизводительный фреймворк для создания веб-приложений на языке Python. Он выступает как ядро системы.

Pydantic - это библиотека для Python для валидации и трансформации данных. Он является центральным инструментом для всех моделей данных  в системе.

- Модели запросов. Для взаимодействия с документами и чатом.

- Модели ответов. Структурирование исходящих данных.

### Хранение и обработка файлов
Можно хранить данные на диске, а в реляционной базе данных хранить пути к файлам, например в PostgreSQL (дата загрузки, оригинальное название документа, идентификатор рекомендации), вектора в PostgreSQL не будут храниться. Получить текст из файлов (они все скачиваются в .pdf) можно с помощью pdfplumber.  

При парсинге важно сохранить логические блоки: заголовки разделов, подразделы. Это позволит точнее ссылаться на источник и поиск будет не по всему документу, а только по его части.

Текст необходимо разбить на небольшие сегменты, обычно по смысловым границам. Предпочтительным будет разделение по ябзацам или разделам. Необходимо убрать лишние переносы строк, лишние пробелы, ненужные символы (например номера страниц из текста, т.к. они сохраняются в тексте при проходе через pdfplumber) без изменения смысла. 

Каждый сегмент должен пройти через модель эмбеддингов, которая его преобразует в числовой вектор. Надо выбрать векторную базу данных, где будут храниться сами вектора и текст (идентификатор рекомендации (некая связь с исходным файлом), номер страницы, название главы или раздела откуда всё бралось).

Векторные базы данных, поддерживающие фильтрацию по методанным, обновление/удаление, масштабирование: Qdrant, Weaviete, Chroma. Векторный поиск не работает на узкоспециализированных терминах. 

### Функциональные требования

1. Пользователь заходит на сайт и имеет возможность написать боту в чат  
2. Пользователь имеет возможность увидеть список документов 
3. Пользователь имеет возможность добавить/удалить пдф документ 
4. Пользователь может посмотреть пдф документ (новая ссылка в браузере)  
5. Пользователю отображается ответ от чат-бота  

Необходим отдельный скрипт для администрирования документов. При добавлении необходимо делать векторизацию и добавить новую информацию в базу данных. При удалении надо убрать все связанные записи из документальной базы. Прямым взаимодействием с документами может заниматься любой пользователь.

## Связь с фронтендом

Основное взаимодействие между фронтендом и бэкендом это добавление и удаление документов, связанных с клиничискими рекомендациями. 

Добавление документа:

    Фронтенд отправляет POST запрос с данными. Бэкенд взаимодейтсвует с Pydantic, принимает файл и возвращает ответ, говорящий о том, что файл принят. Обработка документа.

Удаление документа:

    Фронтенд отправляет DELETE запрос с названием документа. Бэкенд находит все его упоминания в базах данных и файловой системе и удаляет их. Должен вернуться ответ об успешном удалении.

## WebSocket

Вся история диалога храниться в оперативной памяти на бэкенде на время жизни WebSocket соединения. При завершении сессии должна освободиться память, занятая историей чата. Это будет гарантировать отсутствие утечек памяти.

## Контракты

POST /createdoc — загрузка нового документа
GET /doclist — получение списка документов
DELETE /doclist/{doc_id} — удаление документа
GET /doclist/{doc_id}.pdf — скачивание PDF
POST /chat (у тебя там стоит Home в фигме, так что можно и его хз) — отправка сообщения в чат
